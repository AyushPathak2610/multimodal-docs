üí° How It Works
  1. Opens file in write mode
  2. Converts message objects to dictionaries
  3. Writes JSON data to file
  4. File auto-closes (with statement)

Function: load_chat_history_json()

‚ñº
python 
def load_chat_history_json(file_path):
    with open(file_path, "r") as f:
        json_data = json.load(f)
        messages = [
            HumanMessage(**message) if message["type"] == "human" 
            else AIMessage(**message) 
            for message in json_data
        ]
        return messages
‚ú® List Comprehension Magic

This line converts dictionaries back to message objects:

  • Loops through each message in JSON
  • Checks if type is "human" or "ai"
  • Creates appropriate message object
  • **message unpacks dict as keyword arguments

üéØ app.py - Main Application

‚ö†Ô∏è CRITICAL CONCEPT: The Streamlit Execution Loop

This is the most confusing part for beginners!

Streamlit re-runs the ENTIRE script from top to bottom every time:

  • User types something
  • User clicks a button
  • User uploads a file
  • User changes a dropdown

Understanding the Re-run Problem

‚ñº
python - WRONG 
# This code runs EVERY time user interacts
print("Script started")  # Prints on every interaction!
x = 5  # Resets to 5 on every interaction!
python - CORRECT 
# This persists across re-runs
if "counter" not in st.session_state:
    st.session_state.counter = 0  # Only runs once
st.session_state.counter += 1  # Increments each time
‚úÖ Solution: Use st.session_state

Store data in st.session_state to preserve it across re-runs!

Helper Function: clear_input_field()

‚ñº
python 
def clear_input_field():
    if st.session_state.user_question == "":
        st.session_state.user_question = st.session_state.user_input
        st.session_state.user_input = ""
üîí The "Lock" Pattern

This solves the "disappearing input" problem:

  1. User types "Hello" in text box
  2. User presses Enter
  3. on_change event fires
  4. Copies "Hello" to user_question
  5. Clears user_input
  6. Script re-runs
  7. user_input is empty, but user_question has "Hello"

Complete Execution Flow

‚ñº
üìä First Run (New Session)
  1. Script starts
  2. Load config.yaml
  3. Define helper functions
  4. main() called
  5. Initialize session_state (first time only)
  6. Create UI widgets
  7. Load empty history
  8. Create chain
  9. No messages to process
  10. Display empty chat
  11. Save (nothing to save)
  12. Page displays to user
üí¨ User Types Message
  1. User types "Hello" in text box
  2. User presses Enter
  3. on_change event fires BEFORE re-run
  4. set_send_input() called
  5. Script re-runs from top
  6. Check send_input ‚Üí True!
  7. llm_chain.run("Hello")
  8. Gets response: "Hi there!"
  9. Adds both to history
  10. Display chat history
  11. Save to file
  12. Page displays with messages

üêõ Common Issues and Solutions

Issue 1: Messages disappear when I type

‚ñº
‚ùå Wrong
user_input = st.text_input("Type here")
# user_input resets to "" on every re-run!
‚úÖ Right
user_input = st.text_input("Type here", key="user_input")
# Stored in session_state, persists across re-runs

Issue 2: Same message processes multiple times

‚ñº
‚ùå Wrong
if send_button:
    process_message()
# send_button stays True, processes every re-run!
‚úÖ Right
if send_button:
    process_message()
    st.session_state.send_input = False  # Reset flag

Issue 3: Model loads every time I send message

‚ñº
‚ö†Ô∏è Cause

Not caching model loading

‚úÖ Solution
@st.cache_resource
def load_model():
    return create_llm()  # Loads once, cached

def main():
    llm = load_model()  # Fast after first load

‚ö° Performance Optimization Tips

1. Cache Model Loading

    return create_embeddings()

Why: Models load once, stay in memory

2. Lazy Load Chains

‚ñº
python 
# Don't load chain until needed
if st.session_state.user_question:
    llm_chain = load_chain(...)
    llm_chain.run(...)

Why: Saves time when just displaying UI

3. Limit History Display

‚ñº
python 
# Show only last 20 messages
for message in reversed(chat_history.messages[-20:]):
    st.chat_message(message.type).write(message.content)

Why: Faster rendering with long conversations
